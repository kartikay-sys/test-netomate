# NetoMate Flow AI — Architecture v2 Detail

This document outlines the design and implementation of the NetoMate Flow AI v2 engine. The goal of this architecture is to provide high-accuracy suggestions with minimal latency and significantly reduced LLM costs by using a layered, constraint-driven approach.

---

## 🏗️ The 6-Layer Engine

The system processes every query through a series of specialized layers, moving from simple rule-based detection to complex hybrid scoring.

### Layer 0: Input Parser
*   **Role**: Prepares the raw user input for the engines.
*   **Action**: 
    - Identifies if the current flow is "UI" (functions starting with `epcems_`), "Mobile" (radio-level functions), or "Hybrid".
    - Counts the current steps to determine if it's a "Cold Start".
    - Strips and normalizes the NLP query.

### Layer 1: Intent Classifier (Regex/Keyword)
*   **Role**: Routes the query to the correct execution path.
*   **Logic**: A zero-ML, high-speed keyword scanner.
*   **Intents**:
    1.  **Cold Start**: Triggered if `step_count == 0`. Uses frequency-based starting points.
    2.  **Probabilistic**: Triggered by keywords like "next", "continue", "sequence". Uses deterministic math engines.
    3.  **Reasoning**: Triggered by keywords like "improve", "validate", "optimize". Uses math engines + LLM reranking.
    4.  **Mid-Action**: Triggered by keywords like "capture now", "during call". Forces bias towards EPC EMS tasks and suppresses standard sequence.

---

## 🚦 Flow State Machine

Tracks execution phase of the current flow in real time.

States:
*   **EMPTY**: `step_count == 0`
*   **SETUP**: steps present, no `mo_*` or `epcems_capture_*` yet
*   **MID_ACTION**: `mo_mt_*_call_start` seen, no matching terminate
*   **POST_ACTION**: core action completed, capture may be stopped
*   **TEARDOWN**: `pcap_parser` seen, only cleanup steps remain

State is passed as a hard constraint to E2, E3, and the Combiner. Each engine filters its candidates based on what is appropriate for the current state.

---

## ⚙️ Layer 2: Core Engines (The "Heavy Lifters")

Instead of one LLM, we use three specialized engines to generate candidates.

### 1. Engine 1 (E1): Start Predictor
*   **Method**: Frequency analysis of the first step in all benchmark flows.
*   **Best For**: When the user hasn't added any functions yet.
*   **Logic**: It doesn't just look at the global most common function; it looks at the most common *starting* function for the detected category.

### 2. Engine 2 (E2): Markov Chain Engine
*   **Method**: Order-2 Markov Chain with Backoff.
*   **How it Works**: 
    - It analyzes sequences in the dataset as transitions (A → B → C).
    - It calculates the probability of function `C` appearing, given that `A` and `B` just happened.
    - **Backoff Strategy**: If it hasn't seen the pair `(A, B)`, it "backs off" to look at just `(B)`. If it hasn't seen `B`, it looks at global frequency.
    - **Laplace Smoothing**: It adds a small value (pseudocount) to every possible transition to ensure no function has a 0% probability, allowing for "creative" path discovery even for unseen sequences.

### 3. Engine 3 (E3): Embedding Retriever
*   **Method**: Vector Similarity (Semantic Analysis).
*   **Best For**: Queries that describe a concept rather than a sequence (e.g., "how do I test handover?").
*   **Logic**:
    - Converts the entire current flow into a text string and embeds it.
    - Finds the top-k most similar flows in the dataset (using Cosine Similarity).
    - Extracts the functions that follow in those similar flows.
    - **Hybrid Power**: Even if you follow a new path not in the Markov chain, E3 finds flows that *looked* like yours and suggests what they did next.

## ⚖️ The Weighted Score Combiner

The final ranking is determined by a hybrid formula:

$$score(fn) = \alpha \cdot freq\_score + \beta \cdot sim\_score + \gamma \cdot click\_weight$$

*   **$\alpha$ (Frequency)**: How often this function appears globally.
*   **$\beta$ (Similarity)**: How well it fits the immediate sequence or semantic intent.
*   **$\gamma$ (Feedback)**: How often users have clicked this specific suggestion in the past (learned behavior).

---

## 🤖 Layer 3: Micro-LLM Reranker

We use **Gemini 2.5 Flash** as a surgical tool, not a generator.

### When is the LLM used?
The LLM is only invoked for **< 10% of queries** — specifically those classified as **Reasoning**. If a user asks "How do I make this test more robust?" or "make this production ready", the math engines provide valid next steps, and the LLM re-sorts them based on architectural best practices.

### What is sent to the LLM?
We don't send the entire dataset. We send "Compressed Context":
1.  The Current Flow.
2.  The 3-5 candidates generated by the math engines.
3.  The 3 most similar flows found by E3 (as reference).

### Intent Classifier Keywords

| Intent | Keywords |
|--------|----------|
| **Probabilistic** | next, continue, after, follow, what comes, sequence, then, proceed, remaining |
| **Reasoning** | improve, validate, robust, optimize, enhance, add, suggest, better, strengthen, harden, verify, check, quality, coverage, production, ready, missing, complete, secure, reliable, automate, end-to-end |
| **Mid-Action** | capture now, right now, during, while, active, live, running, currently, real-time, mid-call |

## 🛡️ Positional Constraint (Teardown Filter)

A key safety mechanism added after validation testing. The embedding retriever can sometimes pull **teardown functions** (`disconnect_mobile`, `print_results`, `epcems_logout`) prematurely when a flow is still in its setup or mid-execution phase.

### How it works:
1.  Check if any **core test action** has occurred (functions starting with `mo_` or `epcems_capture_`).
2.  If NO core action exists yet → the flow is still in setup → **filter out teardown functions** from E3 results.
3.  If a core action HAS occurred → teardown is legitimate → allow all suggestions through.

This prevents the system from suggesting "disconnect" when you haven't even run a test yet.

---

## 🔄 Layer 4: Feedback Learning Loop

Every time a user clicks a suggestion, the script:
1.  Appends the selection to the flow.
2.  POSTs to `/api/feedback`.
3.  Updates `feedback.json` by incrementing the "weight" of that function.
4.  The next time that function is suggested, its **$\gamma$** weight will be higher, naturally pushing it to the top of the list.

## 🚀 Performance Comparison

| Metric | v1 (LLM Only) | v2 (Layered) |
|--------|---------------|--------------|
| **Latency** | 2.5s - 5.0s | **150ms - 300ms** |
| **LLM Token Usage** | 100% of queries | **~8% of queries** |
| **Deterministic** | No (Hallucinations possible) | **Yes (Constraint-driven)** |
| **Learning** | No | **Yes (Click-tracking)** |
