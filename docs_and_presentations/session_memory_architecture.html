Session Memory

How NetoMate remembers what the engineer did during a session — and why it makes the AI smarter without wasting tokens.

NetoMate Flow AI v3 · April 2026
01
The Problem — AI with Amnesia
Today, every query the test engineer submits is treated as a brand-new, isolated request. The AI has no memory of what happened 30 seconds ago. This creates three concrete pain points:
🔁
Repetitive Suggestions
The engineer ignores epcems_login on Query 3. On Query 4, the AI suggests it again — it has no idea the user already said no. The engineer sees the same function twice and loses trust.
🎯
No Contextual Understanding
After 5 queries and 8 accepted mobile functions, the AI still doesn't know the user is building a voice call test. It suggests random UI functions alongside mobile ones, diluting the relevance of every result.
↩️
Wasted Interactions
Every click the user makes — accept, ignore, query — is valuable signal about their intent. Today, all of this signal is discarded. The 7th query is no smarter than the 1st.

03
Why Not Just Pass the Full Log to the LLM?
The naive approach — dump the entire session timeline into the prompt — is dangerous. The prompt already contains the full 50-flow dataset (~2,000 tokens). Adding raw session logs creates two problems:
📈
Context Window Bloat
A 20-minute session generates 15–25 timeline entries with timestamps, HTML tags, and verbose text. That's 500–800 extra tokens per query. Over a session, the prompt grows uncontrollably — increasing latency and cost.
+500–800 tokens per query growing over session
🌀
Hallucination Risk
LLMs are unreliable at following negative constraints like "don't suggest epcems_login." Adding more text makes them more likely to hallucinate or ignore instructions. The model may see the function name in the session log and re-suggest it precisely because it appeared in context.
negative constraints unreliable name exposure → re-suggestion
~2,000
Existing prompt tokens (dataset)
+800
Raw log injection cost
+15
Our approach (micro-hint)
04
The Two-Layer Architecture
The core insight: 80% of session memory value can be applied in pure Python code — no LLM tokens needed. We split the work between two layers, giving each job to whoever does it best.
⚙️
Layer 1 — Code-Level Signals (Zero LLM Tokens)
Deterministic operations applied in the existing post_process() pipeline. 100% reliable, zero cost, no hallucination possible.
Signal Source Action
Suppression Functions the user ignored Filter out from results before returning
Boost Functions the user accepted Increase confidence score of similar functions
Category lock User consistently accepts mobile functions Override category detection to "Mobile"
0 tokens 100% reliable runs in post_process()
🧠
Layer 2 — Micro-Hint to LLM (~15 Tokens)
A single-sentence behavioral fingerprint appended to the prompt. Gives the LLM just enough context to understand what the user is building — without overwhelming it with raw data.
Existing prompt lines (unchanged)
CURRENT FLOW: ["connect_mobile", "mo_voice_call_start", "capture_device_health"]
FLOW CATEGORY: Mobile
USER QUERY: "improve this test"

↓ Only this line is added
SESSION HINT: Building mobile voice call flow. Recent intent: add validation.
~15 tokens behavioral summary only no raw timestamps or logs
Why this split?
LLMs are bad at "don't suggest X" — they hallucinate anyway. But they're good at "user is building a voice call test, suggest relevant steps." So we give filtering to code and understanding to the LLM.

06
Smart Suppression — Not a Permanent Blacklist
A function suppressed because the user ignored it isn't banned forever. Suppression breaks automatically when the context changes — ensuring the user can always access any function when they actually need it.
🔄
Context-Aware Reset Rules
Suppression is cleared when any of these conditions are met:
Condition Example What happens
Query explicitly asks for that category User types "add EMS monitoring" epcems_* suppression cleared — user is now asking for EMS functions
Flow category changes Flow was Mobile, user adds epcems_login manually → now Hybrid All suppressions cleared — context has fundamentally shifted
Session cleared User clicks "Clear Session" button Everything resets — clean slate
Tab closed Session ends naturally sessionStorage cleared — no permanent bans

09
Hard Boundaries
Session memory is a hint layer — it never overrides the core safety constraints that already exist in post-processing.
Constraint Enforced by Can memory override it?
Vocabulary validation (no hallucinated function names) post_process() — code Never
Category boundary (UI-only ≠ mobile) post_process() — code Never
Duplicate suppression (no already-in-flow functions) post_process() — code Never
Confidence threshold (>5%) post_process() — code Never
Suggestion relevance and ranking LLM reasoning Yes — via micro-hint (15 tokens)
memory = hint only constraints always enforced in code resets on tab close
10
Storage & Privacy
Session memory is entirely client-side. The raw timeline lives in the browser's sessionStorage — it never touches a database, never persists after the tab closes, and never leaves the user's machine except as a distilled micro-hint during the API call.
client-side only sessionStorage scoped no database resets on tab close raw log never sent to LLM
