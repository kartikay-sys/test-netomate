NetoMate Token Architecture

Optimization Plan & Context Reduction Strategy

Core Objective

Reduce overall LLM token consumption while maintaining suggestion quality. Analysed on Architecture v3:

  • Reduce bloated Input Tokens (~4,080) generated by dataset serialization.
  • Minimize strict Output Tokens (~260) driven by over-explanatory JSON schemas.
  • Target a 78% reduction in total tokens per inference.

1. Current Token Flow (Arch v3)

Understanding how "Full-Dataset Injection" is severely bloating our requests.

Input Aggregation Pipeline

System Rules: Base telecom testing assistant instructions constraints.

Pre-processing: Extracted intent flags, temporal session hints, and parsed query signals.

Dataset Payload (The Bloat): Every single request serializes and injects all 50 benchmark flows, demanding massive reasoning to scan the list.

Output Generation Schema

Format: Strict JSON Schema generation natively bound to a set format.

Payload: Exactly 5 function suggestions calculated per request.

Formatting Overhead: Requesting heavy narrative "explanations" for every single candidate, even ones the user never picks.

4. Key Bottlenecks & Inefficiencies

O(N) Scaling of Dataset Context

As more telecom test flows are added to the benchmark dataset, input tokens increase linearly. Providing all 50 complete flow trees is overkill when the LLM only realistically needs 3-5 structurally similar references to construct valid inference.

Unoptimized JSON Schema & Verbose Explanations

The LLM is prompted to produce lengthy human-readable explanations for all 5 suggestions alongside deeply descriptive JSON keys (e.g., "explanation", "confidence"). The user interface typically only scrutinizes the rationale of the top choice.

Stateless Invocations

End-to-end processing of static constraints (System Base) is re-run completely cold for every single query, instead of leveraging modern API Prompt Caching boundaries.

5. Reduction Strategy & Savings Matrix

Phase 1: Output Minification

  • Short Keys: Replace long keys with compressed aliases: {"fn":"...", "e":"...", "c":90}. Mapped locally in Python.
  • Targeted Rationale: Prompt LLM to omit explanations for indices 2-5, retaining only the logic for the primary recommendation.

Phase 2: Selective Retrieval (RAG)

  • Similarity Matching: Implement a lightweight BM25 or nearest neighbor lookup directly in the preprocessing logic.
  • Reduce Scope: Inject only the Top 5 most structurally matched benchmark flows instead of the complete 50.
